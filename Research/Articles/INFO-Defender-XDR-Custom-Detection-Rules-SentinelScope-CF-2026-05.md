## What It Is

Comprehensive guide to Microsoft Defender XDR custom detection rules as of May 2026,
covering the series of platform changes over the past 18 months. Key changes relevant
to current detection engineering work:

- **October 2024:** Near-real-time (NRT) detections entered preview for MDE tables
- **October 2025:** Custom detections became the default for new rules across both
  Defender XDR and Sentinel
- **January 2026:** NRT support expanded to Sentinel data (preview)
- **April 2026:** `SentinelScope_CF` column requirement added — **breaking change for
  scoped workspaces**

---

## The SentinelScope_CF Issue — Actionable Now

This is the most important item. As of April 10, 2026, any custom detection or analytics
rule running in a scoped Sentinel workspace must project `SentinelScope_CF` in the query
output, or alerts generated by that rule will be hidden from scoped analysts.

**Scoping applies if any of these are true:**
- MSSP with scoped access to your workspace
- Regional SOC teams that only see their own data
- Different access tiers (Tier 1 vs Tier 2) with different data visibility

**Key caveats:**
- Scoping only applies to newly ingested data — the 30-day initial scan on rule creation
  uses historical unscoped data, so behaviour may differ at creation vs. steady state
- XDR tables (DeviceProcessEvents, DeviceNetworkEvents etc.) do not directly support
  Sentinel scoping — test in your own tenant if using hybrid rules
- Not yet confirmed whether this affects the current environment specifically

**How to add it if needed:**
```kql
| project ..., SentinelScope_CF
```

Or check if your workspace uses scoping: Sentinel workspace → Settings → Workspace
scoping, or check if any analysts report missing alerts from existing rules.
